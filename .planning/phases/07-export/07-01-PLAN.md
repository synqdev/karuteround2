
Create `public/fonts/` directory. Download Noto Sans JP static TTF files. Use curl to download directly from Google Fonts API:
```
mkdir -p public/fonts
curl -L "https://github.com/google/fonts/raw/main/ofl/notosansjp/NotoSansJP%5Bwght%5D.ttf" -o /tmp/NotoSansJP-variable.ttf
```

IMPORTANT: The above downloads the variable font which react-pdf CANNOT use. Instead, download the static weight files. The static weight TTFs are available from the Google Fonts github repo under the `static/` path. If direct download URLs do not work, check the google/fonts GitHub repo for the correct paths. As a fallback, prompt the user to manually download from https://fonts.google.com/noto/specimen/Noto+Sans+JP and copy the static/ subfolder TTFs.

The two files needed are:
- `public/fonts/NotoSansJP-Regular.ttf` (~4MB)
- `public/fonts/NotoSansJP-Bold.ttf` (~4MB)

After placing font files, verify they exist and are non-trivially sized (should be >1MB each). Add `public/fonts/*.ttf` to `.gitignore` if fonts are too large for the repo, OR commit them (8MB total is acceptable for most repos). Use your judgment based on existing .gitignore patterns.

  <action>
Create `src/components/karute/KarutePdfDocument.tsx` -- a server-only react-pdf Document component (NO 'use client' directive, NO React hooks):

1. Import `Document, Page, Text, View, StyleSheet, Font` from `@react-pdf/renderer` and `path` from `path`.
2. Register Noto Sans JP at MODULE LEVEL (outside any function) using `Font.register()` with `path.join(process.cwd(), 'public', 'fonts', 'NotoSansJP-Regular.ttf')` for normal weight and the Bold variant for bold weight. Family name: `'NotoSansJP'`.
3. Define styles using `StyleSheet.create()`: A4 page with 40pt padding, NotoSansJP font family, 10pt base font size, 1.6 line height. Heading at 16pt bold, subheading at 12pt bold with bottom border, entry containers with light background and rounded corners, category labels at 9pt bold in gray.
4. Export `KarutePdfDocument` component accepting `{ karute: KaruteWithRelations }` (import the type from `@/lib/supabase/karute` or `@/types/karute` -- match whatever Phase 4 established).
5. Layout structure:
   - Page title: customer name + " -- カルテ"
   - Date line: `created_at` formatted with `toLocaleDateString('ja-JP')`
   - "AI サマリー" section with `karute.summary`
   - "エントリー" section: map over `karute.entries`, each in a container showing category label (bold, smaller), content text, and source_quote in gray italic if present
   - "トランスクリプト" section with `karute.transcript`

Create `src/app/api/karute/[id]/export/pdf/route.ts`:

1. Add `export const runtime = 'nodejs'` at top (prevents Edge Runtime where fs is unavailable).
2. Import `renderToStream` from `@react-pdf/renderer`, `createClient` from `@/lib/supabase/server`, `getKaruteRecord` from `@/lib/supabase/karute`, `KarutePdfDocument` from `@/components/karute/KarutePdfDocument`.
3. Export async `GET` handler:
   a. Extract `id` from route params (`const { id } = await ctx.params`).
   b. Auth check: `const supabase = await createClient()`, get user, return 401 Response if no user.
   c. Fetch karute: `const karute = await getKaruteRecord(id)`, return 404 if not found.
   d. Render PDF: `const stream = await renderToStream(<KarutePdfDocument karute={karute} />)`.
   e. Convert Node readable stream to Web ReadableStream using the standard pattern (stream.on data/end/error -> controller enqueue/close/error).
   f. Build filename: `karute-{customerName}-{date}.pdf` (sanitize customer name, use ISO date).
   g. Return Response with Content-Type `application/pdf` and Content-Disposition `attachment; filename="..."`.

Use the RouteContext typing pattern from the research: `ctx: RouteContext<'/api/karute/[id]/export/pdf'>`. If RouteContext is not available in the project's Next.js version, use the standard `{ params: Promise<{ id: string }> }` pattern instead.
  </action>
