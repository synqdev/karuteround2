  <action>
1. **Create `src/components/staff/StaffSwitcher.tsx`** (`'use client'`):
   - Props: `staffList: { id: string; name: string }[]`, `activeStaff: { id: string; name: string } | null`
   - Renders a shadcn/ui `DropdownMenu` with:
     - Trigger: Button showing the active staff name (or "Select Staff" if null)
     - Content: List of all staff members from `staffList`
     - Each item calls `setActiveStaff(staff.id)` Server Action on click
   - Visual indicator (checkmark or bold) on the currently active staff member
   - If `staffList` is empty, show a disabled button with text "No Staff" (edge case — user hasn't created any staff yet)
   - Dark theme styling

2. **Update `src/app/(app)/layout.tsx`** (Server Component):
   - **Read the existing layout first** — do not overwrite existing content. Add staff switcher to the header area.
   - Import `getStaffList` from `@/lib/staff` and `getActiveStaffId` from `@/lib/staff`
   - Fetch staff list and active staff ID server-side
   - Resolve the active staff object: find matching staff from list by cookie ID. If no cookie or cookie ID not found in list, fall back to first staff member alphabetically (`staffList[0]` since list is ordered by name).
   - **Auto-select on first load:** If no `active_staff_id` cookie exists but staff members exist, call `setActiveStaff(staffList[0].id)` to set the cookie for future navigations. This avoids the "no staff selected" pitfall on first use.
   - Pass `staffList` and resolved `activeStaff` as props to `<StaffSwitcher />`
   - Position the switcher in the header area (top right or alongside existing header content)

**Key architectural points:**
- The layout fetches data server-side and passes to the client component — the client component never fetches directly
- `setActiveStaff` writes a cookie + calls `revalidatePath('/', 'layout')`, which re-renders the layout and updates the displayed name
- This pattern means karute save flows (Phase 4) can call `getActiveStaffId()` server-side and always get the correct staff ID from the cookie — no client-supplied staff_id needed

**Staff attribution (STAFF-03) note:** This plan does NOT modify karute save flows — that's Phase 4's responsibility. This plan ensures the cookie mechanism exists and the header displays the active staff. Phase 4 save flows should import `getActiveStaffId()` from `@/lib/staff` and read `staff_id` from the cookie, never from client form fields.
