1. **Read the Phase 1 migration file** to confirm the `profiles` table schema (column names, types). Adapt types accordingly. If Phase 1 hasn't been executed yet, use the schema from 05-RESEARCH.md (id UUID, name text, display_info text, created_at timestamptz).

2. **Create `src/lib/validations/staff.ts`:**
   - Zod schema `staffProfileSchema` with:
     - `name`: string, min 1 ("Name is required"), max 100
     - `display_info`: string, max 500, optional
   - Export `StaffProfileInput` type inferred from schema

3. **Create `src/lib/staff.ts`:**
   - `getStaffList()`: Query `profiles` table, select `id, name, display_info, created_at`, order by `name` ascending. Return array (empty array on error).
   - `getStaffById(id: string)`: Query `profiles` by id, select `id, name, display_info`. Return single record or null.
   - `getActiveStaffId()`: Read `active_staff_id` cookie via `await cookies()` from `next/headers`. Return cookie value or null.
   - All functions use `createClient` from `@/lib/supabase/server`.

4. **Create `src/actions/staff.ts`** with `'use server'` directive:
   - `createStaff(data: StaffProfileInput)`: Validate with `staffProfileSchema`, insert into `profiles`, call `revalidatePath('/settings')`. Throw on error.
   - `updateStaff(id: string, data: StaffProfileInput)`: Validate, update row by id, `revalidatePath('/settings')`. Throw on error.
   - `deleteStaff(id: string)`: Delete row by id, `revalidatePath('/settings')`. Throw on error. (No cascade check in v1 â€” research recommends ON DELETE SET NULL on FK.)
   - `setActiveStaff(staffId: string)`: Write `active_staff_id` cookie via `await cookies()`. Cookie settings: `httpOnly: false` (readable by JS for UI), `sameSite: 'lax'`, `path: '/'`, `maxAge: 60 * 60 * 24 * 30` (30 days). Call `revalidatePath('/', 'layout')` to re-render the header.

**Important:** The `setActiveStaff` action must NOT accept untrusted client data for save operations. It only writes a cookie. All karute save flows must read `staff_id` from the cookie via `getActiveStaffId()`, never from client form fields.

**If no existing RLS policies for profiles, create all four:**

```sql
-- Enable RLS on profiles if not already enabled
alter table profiles enable row level security;

-- SELECT: Any authenticated user can read all staff profiles
-- (single business auth â€” all staff share one login)
create policy "Authenticated users can read all profiles"
on profiles for select
to authenticated
using (true);

-- INSERT: Any authenticated user can create staff profiles
create policy "Authenticated users can insert profiles"
on profiles for insert
to authenticated
with check (true);

-- UPDATE: Any authenticated user can update any profile
create policy "Authenticated users can update profiles"
on profiles for update
to authenticated
using (true)
with check (true);

-- DELETE: Any authenticated user can delete a profile
create policy "Authenticated users can delete profiles"
on profiles for delete
to authenticated
using (true);
```

**Key:** The `using (true)` pattern is correct because v1 uses a single shared business auth account. All authenticated requests come from the same business account, so any authenticated user = the business. Per-staff permissions are out of scope (v2).
